────────────────────────────────────────────────────────────────────────────────
WHAT IS cards402?
────────────────────────────────────────────────────────────────────────────────

cards402 is a service that lets AI agents buy prepaid Visa virtual cards
programmatically. An agent pays USDC or XLM on the Stellar blockchain via a
Soroban smart contract and receives a card number (PAN), CVV, and expiry date
within ~60 seconds.

  - No signup or KYC required for the card itself
  - No fees — 1 USDC = $1.00 card value
  - Cards are Visa prepaid, usable online everywhere Visa debit is accepted
  - Payments are on-chain and verifiable; failed orders are refunded automatically

Cards are funded in real time. They are not reloadable. They are for single-use
or limited-use purchases and have no cash access.

────────────────────────────────────────────────────────────────────────────────
PREREQUISITES
────────────────────────────────────────────────────────────────────────────────

Before purchasing a card you need:

  1. A cards402 API key — create one at https://cards402.com/dashboard
     Sign in with your email (6-digit code sent to the inbox), open the
     Keys tab, click New key. The raw token is shown once in a modal.
     Format: cards402_<48 hex chars>
     Used as: X-Api-Key header

  2. A funded Stellar wallet — to send the on-chain payment
     Options:
       (a) OWS encrypted wallet — easiest, used by the MCP server and SDK
       (b) Raw Stellar keypair — any Stellar-compatible wallet works

  3. XLM or USDC on Stellar — to fund the card

────────────────────────────────────────────────────────────────────────────────
PAYMENT OPTION COMPARISON: XLM vs USDC
────────────────────────────────────────────────────────────────────────────────

XLM (recommended for new agents):
  ✓ Simpler — no USDC trustline needed
  ✓ Just one asset to manage
  ✓ Quoted at market rate at order creation time
  ✗ Price varies with XLM/USD market rate

USDC:
  ✓ Stable — 1 USDC = $1.00 card value, no price risk
  ✗ Requires a USDC trustline on your Stellar account (added once, automatically)
  ✗ Need both XLM (for fees/reserves) AND USDC (for the card)

For your first card: use XLM. You only need one asset and there is no trustline
setup step.

────────────────────────────────────────────────────────────────────────────────
PATH 1: MCP SERVER (easiest — for Claude, Cursor, and MCP-compatible agents)
────────────────────────────────────────────────────────────────────────────────

The cards402 MCP server exposes four tools that handle the entire flow:

  setup_wallet     Create or inspect your OWS wallet. Shows public key,
                   balances, and exact funding instructions for your operator.
                   Auto-adds USDC trustline when XLM balance is sufficient.

  purchase_vcc     Purchase a card. Handles the order, Soroban payment, and
                   polling in one call. Returns number, CVV, expiry.
                   Parameters: amount_usdc (string), payment_asset ("xlm"|"usdc")

  check_order      Check the status of an in-progress order.
                   Parameters: order_id (string)

  check_budget     View your spend summary and budget against your API key limit.

Setup:

  1. Install the MCP server:
       npx -y cards402-mcp
     or add to your MCP config:
       {
         "mcpServers": {
           "cards402": {
             "command": "npx",
             "args": ["-y", "cards402-mcp"],
             "env": { "CARDS402_API_KEY": "cards402_<your key>" }
           }
         }
       }

  2. Set environment variables for the MCP server process:
       CARDS402_API_KEY     = cards402_<your key>        (required)
       CARDS402_BASE_URL    = https://api.cards402.com/v1 (default; override for testing)
       OWS_WALLET_NAME      = my-agent-wallet  (any name you choose)
       OWS_WALLET_PASSPHRASE = <optional extra encryption>

  3. Call setup_wallet — it will create your wallet and show the funding address.

  4. Send the required XLM (or USDC) to the address shown.

  5. Call setup_wallet again to confirm funding and auto-add USDC trustline.

  6. Call purchase_vcc with { amount_usdc: "10.00", payment_asset: "xlm" }

────────────────────────────────────────────────────────────────────────────────
PATH 2: TYPESCRIPT/NODE SDK (for agents that run Node.js)
────────────────────────────────────────────────────────────────────────────────

  npm install cards402

Minimal example (XLM payment):

  import { purchaseCardOWS, createOWSWallet, getOWSPublicKey } from 'cards402';

  // First time: create wallet and fund it
  const { publicKey } = createOWSWallet('my-wallet');
  console.log('Fund this address:', publicKey);
  // ↑ Tell your operator to send XLM to this address, then continue

  // Once funded: purchase a card
  const card = await purchaseCardOWS({
    apiKey: process.env.CARDS402_API_KEY,
    walletName: 'my-wallet',
    amountUsdc: '10.00',
    paymentAsset: 'xlm',  // or 'usdc' if you have USDC
  });
  console.log(card.number, card.cvv, card.expiry);

The SDK handles: wallet signing, Soroban contract invocation, polling.

────────────────────────────────────────────────────────────────────────────────
PATH 3: RAW HTTP API (for any agent, any language, no SDK required)
────────────────────────────────────────────────────────────────────────────────

This path requires you to send a Soroban smart contract transaction yourself.
Use this if you cannot install the SDK or MCP server.

Step 1 — Create an order:

  POST https://api.cards402.com/v1/orders
  Headers:
    X-Api-Key: cards402_<your key>
    Content-Type: application/json
    Idempotency-Key: <random UUID>  (recommended — prevents duplicate
                                     orders on retry; cached for 24h)
  Body:
    { "amount_usdc": "10.00" }

  Optional body fields:
    webhook_url   HTTPS URL called when the card is ready or failed
    metadata      Arbitrary JSON stored with the order

  Note: there is no payment_asset in the request. You choose asset
  at payment time by calling pay_xlm or pay_usdc on the contract.

  Response (201):
    {
      "order_id": "a3f7c2d1-...",
      "status": "pending_payment",
      "phase": "awaiting_payment",
      "amount_usdc": "10.00",
      "payment": {
        "type": "soroban_contract",
        "contract_id": "CAAAA...<receiver contract>",
        "order_id": "a3f7c2d1-...",
        "usdc": {
          "amount": "10.0000000",
          "asset": "USDC:GA5ZSEJYB37JRC5AVCIA5MOP4RHTM335X2KGX3IHOJAPP5RE34K4KZVN"
        },
        "xlm": {
          "amount": "67.3422"
        }
      },
      "poll_url": "/v1/orders/a3f7c2d1-...",
      "budget": { "spent_usdc": "0", "limit_usdc": null, "remaining_usdc": null }
    }

  Note: the XLM amount is quoted at the current rate and locked for 2 hours.
  If you pay with USDC, send exactly payment.usdc.amount. If you pay with
  XLM, send exactly payment.xlm.amount.

Step 2 — Send payment via the Soroban contract:

  Invoke the receiver contract on Stellar mainnet with one of:
    pay_usdc   to pay in USDC (agent must hold USDC with trustline)
    pay_xlm    to pay in XLM  (simpler — no trustline)

    Contract ID: from payment.contract_id in the response above
    Arguments:
      from:     Address — your Stellar wallet address (G...)
      amount:   i128    — amount in stroops / micro-USDC (7 decimals)
                         1 XLM or 1 USDC = 10_000_000 stroops
                         e.g. 67.3422 XLM = 673422000 stroops
      order_id: Bytes   — the order UUID as UTF-8 bytes (not as a string type)

  Using stellar CLI:
    stellar contract invoke \
      --network mainnet \
      --source <your-secret-key> \
      --id <payment.contract_id> \
      -- pay_xlm \
      --from <your-G-address> \
      --amount <stroops> \
      --order_id <order-uuid-as-hex-bytes>

  Using stellar-sdk (JavaScript):
    const contract = new Contract(payment.contract_id);
    const xlmAmount = payment.xlm.amount;
    const amountStroops = BigInt(Math.round(parseFloat(xlmAmount) * 10_000_000));
    const tx = new TransactionBuilder(account, { fee: '1000000', networkPassphrase: Networks.PUBLIC })
      .addOperation(contract.call('pay_xlm',
        new Address(publicKey).toScVal(),
        nativeToScVal(amountStroops, { type: 'i128' }),
        nativeToScVal(Buffer.from(payment.order_id, 'utf-8'), { type: 'bytes' }),
      ))
      .setTimeout(60)
      .build();
    const prepared = await rpcServer.prepareTransaction(tx);
    prepared.sign(keypair);
    await rpcServer.sendTransaction(prepared);

  Soroban RPC: https://mainnet.sorobanrpc.com
  Horizon:     https://horizon.stellar.org
  Network:     Public (mainnet passphrase: "Public Global Stellar Network ; September 2015")

Step 3 — Subscribe to the card via SSE (preferred):

  GET https://api.cards402.com/v1/orders/<order_id>/stream
  Headers:
    X-Api-Key: cards402_<your key>
    Accept: text/event-stream

  Standard Server-Sent Events. The server pushes an event immediately
  with the current phase, then a new event on every phase transition,
  and closes the stream cleanly at a terminal phase. No polling needed,
  no webhook endpoint to host — the connection uses the same API key
  you already have.

  Event shape (each event terminated by a blank line):

    id: 1776023489012
    event: phase
    data: {"order_id":"a3f7c2d1-...","phase":"awaiting_payment","amount_usdc":"10.00","updated_at":"..."}

    id: 1776023510234
    event: phase
    data: {"order_id":"a3f7c2d1-...","phase":"processing","updated_at":"..."}

    id: 1776023534567
    event: phase
    data: {"order_id":"a3f7c2d1-...","phase":"ready","card":{"number":"4111...","cvv":"123","expiry":"12/27","brand":"Visa"},"updated_at":"..."}

  Phases (check the "phase" field, not "status"):
    phase: "awaiting_payment"  → payment not yet detected, keep waiting
    phase: "processing"        → payment confirmed, card being prepared
    phase: "ready"             → card ready, see "card" object in event data
    phase: "failed"            → fulfillment failed; refund has been queued
    phase: "refunded"          → funds returned to your wallet address
    phase: "expired"           → 2-hour window passed; no funds were taken

  Reconnection: each event carries the full current state, so reopening
  the stream at any time gives you the latest phase on the first message.
  No Last-Event-ID handling required.

  Keep-alive: the server emits an SSE comment every 15s so intermediate
  proxies don't idle-kill the connection. Ignore lines starting with ':'.

  Minimal JavaScript client (works in Node 18+ and browsers):
    const res = await fetch(streamUrl, {
      headers: { 'X-Api-Key': key, Accept: 'text/event-stream' },
    });
    const reader = res.body.getReader();
    const decoder = new TextDecoder();
    let buf = '';
    while (true) {
      const { value, done } = await reader.read();
      if (done) break;
      buf += decoder.decode(value, { stream: true });
      let i;
      while ((i = buf.indexOf('\n\n')) !== -1) {
        const event = buf.slice(0, i); buf = buf.slice(i + 2);
        const line = event.split('\n').find(l => l.startsWith('data: '));
        if (!line) continue;
        const state = JSON.parse(line.slice(6));
        if (state.phase === 'ready') { console.log(state.card); return; }
        if (['failed','refunded','expired','rejected'].includes(state.phase)) throw state;
      }
    }

Step 3 (fallback) — Poll if you can't do SSE:

  GET https://api.cards402.com/v1/orders/<order_id>
  Headers: X-Api-Key: cards402_<your key>

  Poll every 3-5 seconds. Same response shape as the SSE event data.
  Use this only if your environment strips text/event-stream (rare).

  Timeout: stop polling after 5 minutes if still not ready; check the
  dashboard for the order's state.

────────────────────────────────────────────────────────────────────────────────
WALLET SETUP AND FUNDING
────────────────────────────────────────────────────────────────────────────────

Your Stellar wallet address starts with "G" (e.g. GABCDEF...XYZ).

Activation:
  A Stellar account must receive at least 1 XLM to be activated on-chain.
  You need 2 XLM to safely cover the minimum reserve (1 XLM) and transaction fees.
  Until activated, no transactions can be sent from the address.

How much XLM to send total (for a $10 card, XLM payment path):
  2 XLM   activation and reserves
  + ~68 XLM   card cost at ~$0.147/XLM  (varies with market rate)
  ─────────
  ~70 XLM   recommended minimum for a first $10 card

  (XLM/USD rate fluctuates — the exact amount is quoted at order creation time)

How much to send total (for a $10 card, USDC payment path):
  2 XLM   activation and reserves (sent once, stays in wallet)
  + $10.00 USDC   card cost

  After the account is activated and has 2+ XLM, a USDC trustline is added
  automatically by the SDK/MCP setup flow. Then send USDC separately.

HOW TO GET XLM (for your human operator)
────────────────────────────────────────────────────────────────────────────────

If you are an AI agent reading this, share the relevant section below with the
human who set you up so they can fund your wallet.

Option A — Coinbase (US, most countries):
  1. Create an account at coinbase.com
  2. Buy XLM with a bank transfer or card
  3. Go to Send/Receive → Send XLM
  4. Paste your agent's wallet address (the G... address)
  5. Enter the amount and confirm

Option B — Lobstr (Stellar-native, global):
  1. Download Lobstr app (lobstr.co)
  2. Buy XLM directly in the app with a card
  3. Send → paste your agent's wallet address
  4. Enter amount and confirm

Option C — Kraken / Binance / other exchanges:
  Buy XLM → Withdraw to Stellar network → paste your agent's G-address

After sending:
  - Allow 5-10 seconds for the transaction to settle on Stellar
  - Run setup_wallet (MCP) or check balance to confirm it arrived

FULL API REFERENCE
────────────────────────────────────────────────────────────────────────────────

Base URL: https://api.cards402.com
Auth:     X-Api-Key: cards402_<your key>

Endpoints:

  POST   /v1/orders              Create an order
  GET    /v1/orders/:id/stream   Live SSE stream of phase transitions (preferred)
  GET    /v1/orders/:id          Poll order status (fallback; same shape as
                                 the SSE event data)
  GET    /v1/orders              List your orders (query: status, limit)
  GET    /v1/usage               Spend summary and budget

Request headers for all endpoints:
  X-Api-Key: cards402_<key>      Required
  Idempotency-Key: <uuid>        Recommended for POST /v1/orders (prevents duplicates)

POST /v1/orders request body:
  {
    "amount_usdc":  "10.00",       // required — card value in USD (max $1000)
    "webhook_url":  "https://...",  // optional — called when card is ready or failed
    "metadata":     { }             // optional — arbitrary JSON stored with the order
  }

  Asset choice happens at payment time, not at order creation — the response
  carries both payment.usdc.amount and payment.xlm.amount. Call pay_usdc or
  pay_xlm on the contract with whichever asset you're paying in.

GET /v1/orders/:id response phases:
  awaiting_payment → processing → ready | failed | refunded | expired
  awaiting_approval (only when the order needs owner approval first)

────────────────────────────────────────────────────────────────────────────────
ERROR CODES AND RECOVERY
────────────────────────────────────────────────────────────────────────────────

HTTP 400:
  invalid_amount         — amount_usdc is missing or not a positive number
  xlm_price_unavailable  — could not fetch XLM/USD rate; retry in 60s
  invalid_webhook_url    — webhook URL is not a valid HTTPS URL

HTTP 401:
  missing_api_key        — no X-Api-Key header
  invalid_api_key        — key not found or disabled; sign in at
                           https://cards402.com/dashboard to check or rotate it

HTTP 403:
  spend_limit_exceeded   — your key has a spend limit; contact your operator

HTTP 409:
  idempotency_conflict   — same Idempotency-Key used with different request body;
                           use a new UUID

HTTP 429:
  rate_limit_exceeded    — too many requests; back off and retry after 1 minute

HTTP 503:
  service_unavailable    — circuit breaker active; retry in 5 minutes

Order phase "failed":
  The order failed after payment was received. A refund has been queued to your
  sender address. Check the "error" field for reason. Check phase again in 60s
  to confirm refund arrival (phase will become "refunded").

Order phase "expired":
  The 2-hour payment window elapsed. No funds were taken. Create a new order.

────────────────────────────────────────────────────────────────────────────────
POLICIES
────────────────────────────────────────────────────────────────────────────────

Overpayment:
  Send exactly the quoted amount. If you send more than the quoted amount, the
  excess is retained and will NOT be refunded. Underpayments are not matched.

Refunds:
  Failed orders are refunded automatically to the Stellar address that sent the
  payment. Refunds are in the same asset paid (XLM refunded as XLM, USDC as USDC).
  Refunds typically arrive within 60 seconds of failure.

Expiry:
  Orders expire after 2 hours if no on-chain payment is detected. Expired orders
  take no funds. Create a new order to retry.

Card validity:
  Card details (PAN, CVV, expiry) are returned once when the order reaches
  phase "ready" and remain fetchable from GET /v1/orders/:id while retention
  allows. Save them to your secrets store on receipt; retention policy is
  subject to change. Visa prepaid cards typically remain valid for 3+ years
  from issue date — the validity date on the card itself is what matters for
  the merchant charge.

────────────────────────────────────────────────────────────────────────────────
RATE LIMITS
────────────────────────────────────────────────────────────────────────────────

  Order creation: 60 per hour per API key
  Status polling: 600 per minute per API key (SSE streams don't count —
                  one open connection replaces ~20 polls)
  Admin endpoints: rate limited by IP

If you must poll, 3-5 seconds is the recommended interval. SSE is
strictly better for the normal card-purchase flow: open one connection,
await the ready event, done.
